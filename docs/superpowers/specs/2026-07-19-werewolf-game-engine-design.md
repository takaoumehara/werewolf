# 人狼ゲーム・エンジン設計仕様

更新日: 2026-07-19

## 目的

携帯電話だけで遊べる人狼ゲームの進行、秘密情報、役職能力、投票、勝敗判定を、画面やFirebaseから独立した決定論的なドメインエンジンとして実装する。ゲームマスターは「コンピューター」または「人間」を選択でき、どちらも同じルールエンジンを利用する。

## 既存資料との関係

- 役職名・陣営・説明文は `card_viewer.html` の `rolesData` を正本とする。
- 世界内の演出は `world-theme.md` に従う。
- カードの表示仕様は `design/current-card-design.md` に従う。
- エンジンはカードの画像、タイトル、演出文を生成・変更しない。

## 設計原則

1. 同じ初期状態・シード・コマンド列は常に同じ結果になる。
2. クライアントは役職、他人の夜行動、サーバー専用状態を直接書き込めない。
3. すべての状態変更は検証済みコマンドから生成されたイベントで説明できる。
4. 役職固有能力は登録可能なルール定義へ分離し、役職追加でフェーズ本体を改変しない。
5. AIはナレーションと操作案内だけを担当し、勝敗・死亡・役職判定を決定しない。
6. タイマーは演出ではなくサーバー時刻を基準にする。期限後の入力はエンジンが拒否する。

## ゲーム状態

`GameState` はサーバー専用の完全状態であり、クライアントへそのまま返さない。

```js
{
  gameId: string,
  revision: number,
  seed: number,
  gmMode: "computer" | "human_player" | "human_observer",
  phase: "lobby" | "role_reveal" | "night" | "day" | "vote" | "finished",
  round: number,
  deadlineAt: number | null,
  players: {
    [playerId]: {
      id: string,
      displayName: string,
      roleId: string,
      team: "citizen" | "werewolf" | "fox",
      alive: boolean,
      joinedAt: number,
      death: { cause: string, round: number } | null,
      flags: Record<string, unknown>
    }
  },
  roleState: Record<string, unknown>,
  pendingActions: Record<string, Action>,
  pendingVotes: Record<string, string | null>,
  winner: { teams: string[], reason: string } | null,
  history: Event[]
}
```

`roleState` には、影武者のコピー元、タフガイの遅延死、恋人のペア、人狼の子どもの強化など、公開してはいけない値を含める。保存時はFirebaseのサーバー専用領域に置く。

## 公開ビューと秘密ビュー

`toPublicView(state)` は次だけを返す。

- ルームID、フェーズ、ラウンド、期限
- 生存状態、表示名、死亡原因の公開済み情報
- 投票の受付状況（誰が誰へ投票したかは投票終了まで非公開）
- 勝敗発表後の全役職

`toPlayerView(state, playerId)` は公開ビューに加えて対象プレイヤーだけの次を返す。

- 自分の役職・陣営・能力説明
- ゲーム開始時に知るべき共有者・双子・人狼仲間の情報
- 自分が受け取る占い・霊媒・神託の結果
- 自分が提出した夜行動の受付状態

他プレイヤーの役職、秘密結果、未公開投票、夜行動の対象は返さない。

## コマンド契約

```js
dispatch(state, { id, actorId, type, payload, expectedRevision, now })
  => { state: GameState, events: Event[] }
```

`id` は再送時の重複防止キー、`expectedRevision` は楽観的排他制御に使う。不正なコマンドは状態を変更せず、`InvalidCommandError` を返す。

最低限のコマンド:

- `START_GAME`: ロビーを開始し、役職を割り当てる。
- `BEGIN_NIGHT`: 夜を開始する。
- `SUBMIT_NIGHT_ACTION`: 役職能力に応じた対象を提出する。
- `RESOLVE_NIGHT`: 受付済みの夜行動を決定順序で一度だけ解決する。
- `BEGIN_DAY`: 朝の公開情報を確定する。
- `START_VOTE`: 昼の投票を開始する。
- `CAST_VOTE`: 生存者が一票を提出・変更する。
- `RESOLVE_VOTE`: 得票を集計し、同票ルールを適用する。
- `SUBMIT_DEATH_REACTION`: ハンター等の死亡時能力の対象を提出する。
- `END_DAY`: 次の夜、または終了へ進む。
- `FORCE_ADVANCE`: 人間GMだけが期限切れ後に進行を確定する。

すべてのコマンドは、現在フェーズ、操作者の生存状態、能力の有無、対象の有効性を検証する。

## フェーズと決定順序

### 夜

1. 役職交換（奇術師・影武者など）
2. 防御・保護（騎士団、ボディガード）
3. 罠の設置と襲撃の衝突（罠師）
4. 人狼の襲撃と一匹狼の単独襲撃
5. 妖狐など襲撃耐性の適用
6. タフガイの遅延死を確定
7. 恋人・ハンター・人狼の子どもの死亡連鎖を解決
8. 予言、霊媒、神託の個別結果を秘密ビューへ配布
9. 勝敗を判定

同一順位の処理順は、プレイヤーIDをソートした安定順で固定する。クライアントの到着順や乱数に依存しない。

### 投票

- 生存者だけが投票できる。
- 自分への投票は設定で許可・禁止できるが、MVPは許可する。
- 同票はMVPでは処刑なしとする。
- 独裁者は正体を公開する一回限りのコマンドで、その日の処刑対象を決められる。
- 処刑後にハンター、恋人、その他の死亡時能力を解決してから勝敗を判定する。

## 勝敗

- 市民陣営: 生存中の人狼陣営が0人になった時点で勝利。
- 人狼陣営: 生存中の人狼の襲撃可能数が、生存中の市民陣営人数以上になった時点で勝利。
- 妖狐: 人狼または市民の終了条件が成立した時点で生存していれば、主陣営と同時に単独勝利。
- アンドロイド: 自分以外が全員死亡した時点で単独勝利。
- 複数の勝利条件が成立した場合は、`winner.teams` に成立した全陣営を安定順で記録する。

## 役職の段階実装

最初のプレイ可能版で次を完全に実装する。

- 市民
- 人狼
- 予言者
- 騎士団
- 霊媒師
- 共有者

役職レジストリには、カードに存在する残りの役職IDも登録できる形を持たせる。第二段階以降で独裁者、ハンター、タフガイ、恋人、罠師、奇術師、影武者、スパイ、裏切り系、人狼の子ども、一匹狼、神様、アンドロイド、妖狐を追加する。`magician_c` は別能力ではなく奇術師のカードバリエーションとして同じルールIDを参照する。

## Firebaseアダプター契約

エンジン本体はFirebase SDKをimportしない。Cloud Function側に次の薄いアダプターを置く。

1. App Check・認証・ルームメンバーシップを検証する。
2. サーバー専用状態をトランザクションで読み込む。
3. `dispatch` を一度だけ実行する。
4. イベント履歴、完全状態、公開ビュー、各プレイヤーの秘密ビューを原子的に保存する。
5. `commandId` を処理済み台帳へ保存し、再送時は同じ結果を返す。

参加・招待の安全性は `docs/firebase-live-access-key.md` と `snap-pair` の契約に従い、クライアントから権威状態を直接書き換えられないようにする。

## 非目標

- AIによる会話監視・嘘判定・自由なルール生成
- 画像生成やカードレイアウトの変更
- 初版で全役職の曖昧な能力を勝手に決めること
- クライアント時刻やクライアント乱数を勝敗へ利用すること
