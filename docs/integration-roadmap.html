Engine × UI · Integration Roadmap

ゲームエンジンと UI を繋ぐ

エンジンは純粋なESMで、ブラウザでもサーバーでも同じ dispatch で動く。役職IDはカード資産IDと1:1で一致。だから繋ぎ込みの本質は「1つの GameClient インターフェイスを決め、ローカル版→Firebase版と差し替える」こと。以下がその工程と、複数AIセッションでの並行の切り方。

01繋ぎ目の正体 — 1つのインターフェイス、2つの実装

UIから見れば「部屋・プレイヤー・自分の役職・フェーズを購読し、コマンドを送る」だけ。この契約を GameClient として固定すれば、中身をローカルエンジン→Firebaseへ差し替え可能になる。しかも契約の中身(コマンド10種・ビュー2種・役職ID)はエンジン側で既に確定済み。

⑤ UI (mobile_app.html)

画面は「今のビュー」を描き、操作はclient.send(...)を呼ぶだけ。エンジンもFirebaseも知らない。

→
interface GameClient
subscribePublic(cb)
subscribeSelf(uid,cb)
send(command)
createRoom() / join(code)
↙   ↘

LocalGameClient Phase 1

ブラウザ内で createGame/dispatch を直接回す。Firebase不要・オフライン。UI配線の検証用。

FirebaseGameClient Phase 3

callable呼び出し + onValue 購読。同じインターフェイスなのでUIは無改造で差し替え。

★ 契約はもう凍っている
実質的な設計作業はほぼ不要。コマンド= START_GAME BEGIN_NIGHT SUBMIT_NIGHT_ACTION CAST_VOTE RESOLVE_* ほか計10種。ビュー= toPublicView(state)(全員向け・役職を含まない)と toPlayerView(state,uid)(本人の self.roleId 等)。役職ID= ROLE_IDS の25個が {id}_ver_a.png のカード資産名と完全一致 → 変換表いらず。Phase 0 でやるのはこれを1ファイルに書き写して固定することだけ。

02推奨ルート — ローカル先行 → エミュレータ → 実機

いきなりFirebaseに行かない。①ローカルで先にUI×エンジンを繋ぎ、②Firebaseはまずエミュレータで組み、③最後に実機デプロイ。課金・クラウド設定は最終段まで不要で、各段が独立して「動く」状態になる。

0

契約の凍結 半日 · 1セッション逐次

目的: 並行作業を始める前に「唯一の共有面」を固定する。ここだけは分岐できない。

  • game-client/game-client-interface.mjs に GameClient の型(JSDoc)を定義
  • コマンド10種と payload、toPublicView/toPlayerView の返り値形を docs/…/integration-contract.md に転記
  • UIのモック(PLAYERS等)を、このビュー形と同じ shape に整形(後の差し替えを無改造にする布石)
成果物: 共有インターフェイス1本 + 契約書1本(全レーンがこれをimport/参照)
1

ローカル統合 — Firebaseゼロで本物のゲームが回る Lane A

目的: UIとエンジンの結線を、ネットワーク無しで先に完成させる。最大の価値・最小のリスク。

  • LocalGameClient 実装(ブラウザ内で createGame/dispatch、複数プレイヤーはホットシート or 疑似)
  • mobile_app.html のモック描画を toPublicView/toPlayerView 購読へ置換
  • 役職確認・夜行動・投票・処刑・勝敗が実エンジンの状態で動く(カードは既存のまま)
依存: エンジン(済) + Phase0  |  検証: 1台のブラウザで全フロー通し
2

Firebaseバックエンド(エミュレータ上) Lane B

目的: 既存の6タスク・プランを実装。ローカルの Firebase Emulator Suite で回すのでクラウド課金・プロジェクト作成は不要。

  • firebase-backend/: Callable 4本(createSnapRoom/joinSnapRoom/startWerewolfGame/dispatchWerewolfCommand)
  • database.rules.json: public/publicEvents=メンバー読取、privateViews=本人のみ、authoritative=サーバーのみ
  • エンジン+アダプタ契約をFunctions内にbundle。actorIdは必ず auth.uid から(なりすまし防止)
  • firebase.json の emulators 設定 + 契約/ルールの自動テスト
依存: エンジン+アダプタ(済) + Phase0の契約  |  参照: docs/…/firebase-game-session プラン
3

実マルチデバイス — QRで複数スマホ参加 Lane C → 統合

目的: FirebaseGameClient を LocalGameClient と同じインターフェイスで実装し、UIを差し替え。実プロジェクトへデプロイ。

  • game-client/firebase-game-client.mjs(匿名認証 + callable + onValue 購読)
  • QR/6桁コード参加を live_access_key.mjs の鍵ビジュアルに接続
  • 実Firebaseプロジェクト作成 + Blazeプラン + firebase deploy(ここで初めてクラウド設定)
  • 再接続・重複送信・ホスト離脱を実機2〜3台で確認
依存: Phase0の契約 + Lane B(callable名/パス)  |  要ユーザー判断: Firebaseプロジェクトの選定
4

ビジュアル仕上げ Lane D · 全期間並行

目的: 02プロンプトが求めた hi-fi 状態(loading/empty/error/offline/reconnecting)・演出・モーションを詰める。配線に依存しない。

  • 各状態のhi-fi画面 / 処刑・勝敗の cinematic reveal / フェーズ遷移の motion storyboard
  • アクセシビリティ最終監査(360px・200%拡大・reduced-motion)
依存: design-system.css(済)のみ。Claude Design の出番はここ(→ §04)

03複数AIセッションでの並行 — 4レーンの切り方

Phase 0で契約を凍結した瞬間から、A/B/Dは別々のAIセッション(=別ウィンドウ)で同時に進められる。鍵は「各レーンが触るディレクトリを分ける」こと。git worktree で作業コピーを分離すれば衝突しない。

LANE A · FRONTEND

UI × エンジン結線

Phase 1。sonnet級で十分(仕様明確)。

  • LocalGameClient 実装
  • mobile_app.html をビュー購読へ
触る: mobile_app.html / game-client/local-*
始動可: Phase0直後
LANE B · BACKEND

Cloud Functions + ルール

Phase 2。統合判断が要る→標準〜上位モデル。

  • Callable 4本 + emulator
  • database.rules.json + テスト
触る: firebase-backend/
始動可: Phase0直後(UI不要)
LANE C · CLIENT SDK

FirebaseGameClient

Phase 3。Bのパス確定後に統合。

  • 認証 + callable + 購読
  • QR鍵ビジュアル接続
触る: game-client/firebase-*
始動可: 設計書からPhase0直後に着手可 / 統合はB完了後
LANE D · DESIGN

ビジュアル仕上げ

Phase 4。全期間ずっと並行。Claude Design。

  • hi-fi状態・cinematic・motion
  • a11y最終監査
触る: 新規html/css(coreは触らない)
始動可: いつでも(依存なし)

04意思決定ポイントと Claude Design の置き場所

判断1 — 権威モデル(いちばん大事)

推奨 · 設計書と一致

サーバー権威(Cloud Functions)

秘密の役職がFirebaseルールで構造的に守られる

エンジンはサーバーだけ。クライアントは改ざん不能

まずエミュレータで組めば課金・プロジェクト不要で完成できる

Functions実装とデプロイの手間(=Lane B)

高速プロトタイプ

snap-pair(クライアント直結)

Firebase設定ゼロ・即マルチデバイス

役職を隠せない(DBが公開読取)→ 設計書が明確に却下

身内の動作確認どまり。公開リリース不可

おすすめの落とし所
サーバー権威 × エミュレータ先行。Phase 1(ローカル)で「ゲームが回る」体験を先に確定させ、Phase 2 は Firebase Emulator で完成させる。実クラウド(課金・鍵)は Phase 3 の最後だけ。snap-pairは採らないが、その「QR/6桁コードで繋ぐUX」の考え方だけは live_access_key.mjs に既に活きている。

判断2 — Claude Design を使うべき所・使わない所

用途Claude Design理由
hi-fi状態画面(loading/error/offline/reconnecting)向くビジュアル生成が本領。design-system.css のトークンを渡せば一貫
処刑・勝敗の cinematic / motion storyboard向く静止画・演出設計に強い
カードの見せ方バリエーション検討補助的正本(card_viewer)は固定。あくまで案出し止まり
エンジン結線 / Firebase / 状態管理使わないロジック配線はコード作業。Lane A/B/C(=Claude Code)の領分

→ Claude Design は Lane D 専任。design-system.css と mobile_app.html のスクショを入力に、成果は独立HTML/画像で受け取り、Lane A が取り込む。配線タスクには回さない。
